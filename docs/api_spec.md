## 1. API Overview

### 1.1 Base URLs

| Service | URL | Description |
|---------|-----|-------------|
| REST API | `http://localhost:8000/rest/v1` | PostgREST data API (via Kong) |
| Auth | `http://localhost:8000/auth/v1` | GoTrue authentication service |
| Storage | `http://localhost:8000/storage/v1` | File storage service |
| Realtime | `http://localhost:8000/realtime/v1` | Realtime subscriptions |
| Studio | `http://localhost:54323` | Supabase Studio UI |

### 1.2 API Design Principles

The REST API is **fully auto-generated by PostgREST v13.0.7** from the PostgreSQL schema. There are no hand-written route handlers. All resource endpoints correspond to public schema views defined in `docker/volumes/db/init/24-public-api-views.sql`, which expose tables from the domain schemas (`shared`, `trees`, `sensor`, `pointclouds`, `environments`, `imagery`).

Key principles:
- Every public schema view is a readable endpoint; writable views have `INSTEAD OF` triggers routing mutations to the underlying schema tables
- Resources are addressed by lowercase view name (e.g., `trees`, `sensorreadings`, `pointclouds`)
- PostgREST filter operators control query narrowing without custom query parameters
- Row Level Security (RLS) is enforced at the database level for every table
- S3 URI paths are stored for LiDAR point cloud files (`.las`, `.laz`, `.ply`); file content is not served through this API

### 1.3 API Versioning

All REST endpoints are served at `/rest/v1`. There is currently no second API version. Kong routes `/rest/v1/*` to the PostgREST service internally.

## 2. Authentication & Authorization

### 2.1 Authentication Methods

Every request to `/rest/v1/*` requires two headers:

| Header | Value | Purpose |
|--------|-------|---------|
| `apikey` | `<ANON_KEY>` or `<SERVICE_ROLE_KEY>` | Kong consumer identification |
| `Authorization` | `Bearer <JWT>` | PostgREST role resolution via Supabase Auth JWT |

Both keys are defined in `docker/.env` as `ANON_KEY` and `SERVICE_ROLE_KEY`. For anonymous public access, the anon key is used for both headers (it is itself a valid JWT).

**JWT Token Structure (HS256, issued by GoTrue v2.182.1):**

```json
{
  "header": { "alg": "HS256", "typ": "JWT" },
  "payload": {
    "role": "anon",
    "iss": "supabase",
    "iat": 1714000000,
    "exp": 1714086400
  }
}
```

For authenticated users, `role` is `authenticated` and `sub` contains the GoTrue user UUID.

### 2.2 Authorization (Role-Based via RLS)

Row Level Security is enabled on all domain tables. PostgREST maps the JWT `role` claim to a PostgreSQL role:

| Role | Kong Consumer | Access Level |
|------|--------------|-------------|
| `anon` | `anon` (ANON_KEY) | SELECT on all public views |
| `authenticated` | `anon` or `service_role` | SELECT + INSERT + UPDATE + DELETE on data tables |
| `service_role` | `service_role` (SERVICE_ROLE_KEY) | Full access; bypasses all RLS policies |

**RLS patterns in this project** (from `docker/volumes/db/init/20-rls-policies.sql`):
- Reference tables (Species, Locations, VariantTypes, SoilTypes, ClimateZones) â€” SELECT for everyone
- Data tables (Trees, Sensors, SensorReadings, PointClouds, etc.) â€” SELECT for everyone; INSERT/UPDATE/DELETE for `authenticated`
- Lookup tables (SensorTypes, morphology lookups) â€” SELECT for everyone; no INSERT/UPDATE/DELETE through public API

#### Public view categories

The domain data lives in the custom schemas (`shared`, `trees`, `sensor`, â€¦); `public` holds only views. PostgREST serves `public` as its **default profile**, so exposing views there lets clients use simple names (`/trees`) without a schema-selection header. The public views fall into three groups:

1. **Pass-through views** (`24-public-api-views.sql`) â€” 1:1 wrappers over a single domain table (`trees`, `sensors`, `sensorreadings`, `species`, all morphology/lookup tables, â€¦). They exist purely to expose the domain schemas over REST; writable ones carry `INSTEAD OF` triggers.
2. **Composite / export views** â€” `silva_input` (SILVA model export), `growth_simulations` / `simulation_runs` (simulator output).
3. **Unreal Engine views** (`ue_*`) â€” flat, join-free payloads shaped for UE Blueprint HTTP import. `ue_trees` (trees + variant + scenario + species + main-stem DBH, flattened) is defined in `25-forest-state-views.sql`; `ue_sensors` and `ue_sensorreadings` in `28-sensor-views.sql`.

> **`ue_trees`** is the single flat tree endpoint. It replaced the former `forest_state` view (XRFF-240), which was consolidated into `ue_trees` â€” see `33-consolidate-ue-trees.sql`. Filter by `variantid` to load one time step: `GET /ue_trees?variantid=eq.<id>`.

#### Linking trees â†” sensors â†” readings in Unreal Engine

The three `ue_*` views form the complete chain UE needs; join on stable keys:

```
GET /ue_trees?variantid=eq.<v>                              â†’ tree catalogue (treeid, treeentityid)
GET /ue_sensors?linked_tree_entity_id=eq.<treeentityid>     â†’ all sensors on that physical tree
GET /ue_sensorreadings?sensorid=eq.<sensorid>&order=timestamp.desc&limit=96   â†’ that sensor's readings
```

Join sensors to trees by **`linked_tree_entity_id`** (the persistent physical-tree UUID), not `linked_tree_id` (a single variant row) â€” this keeps the link stable across growth variants. `ue_sensors` also carries the latest reading inline (plus `sensor_model` = real instrument and `data_owner`), so a per-tree sensor list needs no extra readings or `/sensors` call. `linked_tree_*` is populated by `scripts/import/link_sensors_to_trees.py` (see [database_schema.md Â§3.9](database_schema.md)); meteo/soil-station sensors have `NULL` tree fields.

`ue_trees` also carries `aquarius_name` and a `has_sensors` boolean, so UE can flag instrumented trees without a second query â€” filter with `GET /ue_trees?has_sensors=eq.true`.

### 3.2 Standard CRUD Methods

| Method | Path | Description | Auth Required |
|--------|------|-------------|---------------|
| GET | `/rest/v1/{resource}` | List / filter records | anon key |
| GET | `/rest/v1/{resource}?{filters}&select={cols}` | Projected and filtered list | anon key |
| POST | `/rest/v1/{resource}` | Create record | authenticated JWT |
| PATCH | `/rest/v1/{resource}?{filters}` | Update matching records | authenticated JWT |
| DELETE | `/rest/v1/{resource}?{filters}` | Delete matching records | authenticated JWT |

**Example: GET trees filtered by location**

Request:
```
GET /rest/v1/trees?locationid=eq.1&datasourcetype=eq.field&select=variantid,treeentityid,height_m,speciesid&limit=50
apikey: <ANON_KEY>
Authorization: Bearer <ANON_JWT>
```

Response (200 OK):
```json
[
  {
    "variantid": 42,
    "treeentityid": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "height_m": 28.5,
    "speciesid": 3
  }
]
```

**Example: POST new tree variant**

Request (POST `/rest/v1/trees`):
```json
{
  "locationid": 1,
  "varianttypeid": 1,
  "speciesid": 3,
  "measurementdate": "2025-09-11",
  "datasourcetype": "field",
  "height_m": 22.4,
  "position": "SRID=4326;POINT(7.85 47.99)"
}
```

Response (201 Created with `Prefer: return=representation`):
```json
[{ "variantid": 101, "treeentityid": "b2c3d4e5-...", "height_m": 22.4, ... }]
```

### 3.3 Sensor Readings (Time-Series Queries)

| Pattern | Query String | Description |
|---------|-------------|-------------|
| Latest N readings | `?sensorid=eq.{id}&order=timestamp.desc&limit=100` | Most recent readings |
| Time range | `?sensorid=eq.{id}&timestamp=gte.{iso8601}&timestamp=lte.{iso8601}` | Bounded time window |
| Quality filter | `?sensorid=eq.{id}&quality=eq.good&order=timestamp.desc` | Good-quality only |
| Active sensors | `?isactive=eq.true&locationid=eq.{id}` | Active sensors at a location |

### 3.4 RPC Functions

SQL functions are exposed at `/rest/v1/rpc/{function_name}` via HTTP POST. All three RPC functions are granted to `anon`, `authenticated`, and `service_role`.

| Function | Endpoint | Source File | Description |
|----------|----------|-------------|-------------|
| `bulk_upsert_sensors` | `POST /rest/v1/rpc/bulk_upsert_sensors` | `22-aquarius-integration.sql` | Upsert sensors from JSON array (conflict on `externalid`) |
| `bulk_insert_readings` | `POST /rest/v1/rpc/bulk_insert_readings` | `22-aquarius-integration.sql` | Bulk insert readings, skip duplicates (unique on `sensorid`+`timestamp`) |

## 4. Request & Response Formats

### 4.1 PostgREST Query Parameters

| Parameter | Syntax | Description | Example |
|-----------|--------|-------------|---------|
| `select` | `col1,col2,col3` | Column projection | `?select=variantid,height_m,speciesid` |
| `order` | `col.asc` or `col.desc` | Sort order | `?order=timestamp.desc` |
| `limit` | integer | Max rows returned | `?limit=100` |
| `offset` | integer | Skip rows (for pagination) | `?offset=200` |

### 4.2 PostgREST Filter Operators

Filters are applied as query parameters in the form `?{column}={operator}.{value}`:

| Operator | Meaning | Example |
|----------|---------|---------|
| `eq` | Equal | `?sensorid=eq.5` |
| `neq` | Not equal | `?quality=neq.bad` |
| `lt` | Less than | `?height_m=lt.10` |
| `gt` | Greater than | `?height_m=gt.30` |
| `gte` | Greater than or equal | `?timestamp=gte.2025-01-01T00:00:00Z` |
| `lte` | Less than or equal | `?timestamp=lte.2025-12-31T23:59:59Z` |
| `like` | Pattern match (case-sensitive) | `?variantname=like.Ecosense*` |
| `ilike` | Pattern match (case-insensitive) | `?scientificname=ilike.*fagus*` |
| `in` | Match any value in list | `?datasourcetype=in.(lidar,field)` |
| `is` | IS NULL / IS TRUE / IS FALSE | `?decommissiondate=is.null` |

### 4.3 Standard Response Structure

PostgREST returns plain JSON arrays. No envelope wrapper is added.

**Success (list):**
```json
[
  { "variantid": 1, "height_m": 28.5, "speciesid": 3 },
  { "variantid": 2, "height_m": 31.2, "speciesid": 3 }
]
```

**Pagination header (returned when `Prefer: count=exact` is sent):**
```
Content-Range: 0-49/1200
```

## 5. Error Codes

### 5.1 HTTP Status Codes

| Status | Meaning | When Used |
|--------|---------|-----------|
| 200 | OK | Successful GET, PATCH, RPC |
| 201 | Created | Successful POST with `Prefer: return=representation` |
| 204 | No Content | Successful POST/PATCH/DELETE without representation |
| 400 | Bad Request | Malformed filter, invalid column name, type mismatch |
| 401 | Unauthorized | Missing or invalid `apikey` / `Authorization` header |
| 403 | Forbidden | JWT valid but RLS policy denies the operation |
| 404 | Not Found | Resource (view) does not exist in `public` schema |
| 406 | Not Acceptable | Invalid `Accept` header |
| 409 | Conflict | Unique constraint violation (e.g., duplicate `sensorid`+`timestamp`) |
| 500 | Internal Server Error | Database error or unhandled constraint violation |

### 5.2 PostgREST Error Response Format

```json
{
  "code": "PGRST116",
  "details": null,
  "hint": null,
  "message": "The result contains 0 rows"
}
```

Common error codes:

| Code | Description |
|------|-------------|
| `PGRST116` | Query returned 0 rows when exactly 1 expected (`Prefer: single`) |
| `PGRST204` | Column does not exist in the view |
| `42501` | RLS policy denied the operation |
| `23505` | Unique constraint violation |
| `23503` | Foreign key constraint violation |
| `23514` | Check constraint violation (e.g., invalid `datasourcetype` value) |

Rate limiting is not configured in the current Kong setup (`docker/volumes/api/kong.yml`). Kong handles routing and API-key auth only. Rate limiting should be added via Kong plugins before any public deployment.

## 7. Pagination

PostgREST uses `limit`/`offset` parameters. Total count is available via `Prefer: count=exact`.

**Request:**
```
GET /rest/v1/sensorreadings?sensorid=eq.5&order=timestamp.desc&limit=100&offset=0
Prefer: count=exact
```

**Response headers:**
```
Content-Range: 0-99/45231
```

Without `Prefer: count=exact`, the total count is omitted for performance. For high-volume tables like `sensorreadings`, avoid `count=exact` in production queries.
